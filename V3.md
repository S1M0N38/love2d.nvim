# V3 — Design Document

A focused but complete Neovim plugin for LÖVE 2D game development.
Targets both LÖVE developers new to Neovim and Neovim users trying LÖVE.

---

## Decisions

- **Philosophy**: Focused but complete. LSP + run/stop + compiler + detection + health. No scope creep beyond this.
- **Audience**: LÖVE devs new to Neovim AND Neovim users trying LÖVE. Zero-config ideal after `setup()`.
- **Neovim minimum**: 0.12.2+ (breaking change, V3.0)
- **Installation docs**: lazy.nvim for now. vim.pack docs deferred.
- **LSP definitions**: Git submodules, moved to `./libraries/` for a clean repo root. No `vim.pack`, no dynamic install.
- **LSP architecture**: Static settings in `lsp/lua_ls.lua` (file-based config). Dynamic library path injection in `config.lua`. Minimal cleanup of existing approach.
- **Public API**: `setup()`, `run()`, `stop()`. `find_src_path()` and `is_love2d_project()` stay on the module as internal-use convention (not removed, but documented as internal).
- **Commands**: `:LoveRun` and `:LoveStop` only. No new commands.
- **Keymaps**: No defaults. Users define their own.
- **Debug window**: Keep as-is. May redesign in a future minor release.
- **Compiler plugin**: `compiler/love.lua` as-is. Errorformat covers `:make` quickfix.
- **Health checks**: 5 checks (did_setup, Neovim version, love binary, lua_ls, treesitter).
- **GLSL injection**: `after/queries/lua/injections.scm` as-is.
- **Test framework**: Migrate to mini.test (via `lazy.minit`). Current e2e runners (`tests/e2e_*.lua`) are temporary and will be replaced.
- **Test priority**: Source changes first, then test migration, then docs.
- **conf.lua errorhandler**: Document as a recommended pattern in vimdoc, not a shipped feature.
- **CI**: Format check + typecheck now. Test job added after mini.test migration.
- **Release criteria**: All steps complete before merging to main.
- **Target environment**: Stock Neovim only. No distribution-specific effort.

## 1. Submodule relocation: `love2d/` → `libraries/`

**Why**: Two directories at repo root with generic names (`love2d/`, `luasocket/`) is confusing. Moving them under `libraries/` makes the repo structure self-documenting and keeps the root clean.

### Before (current)

```
love2d/       ← LÖVE API type definitions (git submodule)
luasocket/    ← LuaSocket type definitions (git submodule)
```

### After (V3)

```
libraries/love2d/    ← LÖVE API type definitions (git submodule)
libraries/luasocket/ ← LuaSocket type definitions (git submodule)
```

### Impact

- `config.lua`: `setup_lsp_libraries()` updates glob pattern from `"love2d"` → `"libraries/love2d"` and `"luasocket"` → `"libraries/luasocket"`
- `.gitmodules`: URL paths updated
- `health.lua`: Could optionally check submodule presence (deferred)
- CI: No impact — submodules are checked out by `actions/checkout@v4` with `submodules: true`

## 2. LSP configuration

### Static settings: `lsp/lua_ls.lua`

File-based LSP config (Neovim 0.11+ pattern). Ships with the plugin, auto-discovered from runtimepath.

```lua
return {
  cmd = { "lua-language-server" },
  settings = {
    Lua = {
      runtime = { version = "LuaJIT" },
      diagnostics = { disable = { "duplicate-set-field" } },
      workspace = { checkThirdParty = false },
    },
  },
}
```

### Dynamic library injection: `config.lua`

Resolves submodule paths at runtime and injects into lua_ls `workspace.library`.

```lua
local libs = { "libraries/love2d", "libraries/luasocket" }
for _, name in ipairs(libs) do
  local path = vim.fn.globpath(vim.o.runtimepath, name, false, true)
  if path and path[1] then
    table.insert(existing, path[1])
  end
end
```

### User customization

Users can override lua_ls settings via their own `after/lsp/lua_ls.lua` — Neovim's built-in config chain handles merging. No plugin-specific escape hatches needed.

## 3. API surface

### Public (documented, stable)

| Function | Description |
|----------|-------------|
| `love2d.setup(opts)` | Initialize plugin. Must be called before run/stop. |
| `love2d.run(path)` | Run LÖVE project at `path`. Empty string auto-detects. |
| `love2d.stop()` | Stop running LÖVE project. |

### Internal (available but not guaranteed stable)

| Function | Description |
|----------|-------------|
| `love2d.find_src_path(path)` | Find directory containing `main.lua`. Used by `:LoveRun`. |
| `love2d.is_love2d_project()` | Detect if cwd is a LÖVE project. Used by `config.lua`. |

### User commands

| Command | Description |
|---------|-------------|
| `:LoveRun [path]` | Run LÖVE project. Auto-detects path if omitted. |
| `:LoveStop` | Stop running LÖVE project. |

### Configuration options

| Option | Default | Description |
|--------|---------|-------------|
| `path_to_love_bin` | `"love"` | Path to LÖVE executable |
| `restart_on_save` | `false` | Auto-restart on Lua file save |
| `debug_window_opts` | `nil` | Window config for debug output split |

### Removed (breaking changes from V2)

| Option | Notes |
|--------|-------|
| `setup_makeprg` | Removed — compiler always set for LÖVE projects |
| `setup_compiler` | Removed — compiler always set for LÖVE projects |
| `identify_love_projects` | Removed — detection always runs |

## 4. Testing strategy

### Framework: mini.test (via lazy.minit)

- `tests/minit.lua` — harness bootstrap
- `tests/love2d_spec.lua` — unit tests (CI-safe, no love binary needed)
- `tests/platform_spec.lua` — love binary tests (skip with `pending()` in CI)
- `tests/lsp_spec.lua` — LSP unit + integration tests

### Current e2e runners are temporary

`tests/e2e_game.lua` and `tests/e2e_bad_game.lua` are stopgap tests that verify the current implementation doesn't regress. They will be replaced by mini.test specs.

### Migration order

1. Complete all source changes first
2. Write mini.test specs against the final API
3. Delete busted specs (`spec/`) and e2e runners
4. Add test job to CI

## 5. conf.lua errorhandler pattern

The `tests/game/conf.lua` includes a custom `love.errorhandler` that writes parseable errors to stderr:

```
file:line: error: message
```

This makes `:make` populate the quickfix list with correct file, line, and severity. V3 will document this as a recommended pattern in vimdoc — users copy it into their project's `conf.lua`. Not auto-injected by the plugin.

## 6. File structure (V3 target)

```
lua/love2d/
  init.lua          — Main module: setup(), run(), stop(), find_src_path()
  config.lua        — Options, LSP library injection, autocmds
  utils.lua         — Shared utilities (is_love2d_project, notify)
  types.lua         — LuaCATS type definitions (@meta)
  health.lua        — :checkhealth love2d support

compiler/
  love.lua          — Compiler plugin (makeprg + errorformat)

lsp/
  lua_ls.lua        — Static lua_ls settings (file-based LSP config)

libraries/          — LÖVE API definitions for LSP (git submodules)
  love2d/           — LÖVE type definitions (LuaCATS/love2d)
  luasocket/        — LuaSocket type definitions (LuaCATS/luasocket)

plugin/
  love2d.lua        — User commands (:LoveRun, :LoveStop)

after/queries/lua/
  injections.scm    — Treesitter injection for GLSL in LÖVE shaders

tests/
  minit.lua         — mini.test runner (lazy.minit bootstrap)
  love2d_spec.lua   — Unit tests (CI-safe)
  platform_spec.lua — Love binary tests (local-only, pending-based)
  lsp_spec.lua      — LSP unit + integration tests
  game/             — Sample LÖVE game for testing
  bad-game/         — Broken LÖVE game for error tests

spec/               — [Deleted in mini.test migration]
tests/e2e_*.lua     — [Deleted in mini.test migration]

doc/love2d.txt      — Vimdoc help file (rewritten at end of V3)
```

## 7. Zero-config experience

After `require("love2d").setup({})` in a LÖVE project:

- ✅ `love` global recognized by lua_ls (hover → docs)
- ✅ LuaSocket types available
- ✅ `:make` runs game and parses errors into quickfix
- ✅ `:LoveRun` / `:LoveStop` commands available
- ✅ `:checkhealth love2d` diagnostics
- ✅ GLSL syntax highlighting in `love.graphics.newShader()` strings

The only thing users need to do:
1. Install the plugin
2. Call `setup({})` (explicitly or via lazy.nvim `opts`)
3. Optionally set `path_to_love_bin` if `love` isn't in PATH
